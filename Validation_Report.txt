ASSUMPTIONS (where milestone docs were ambiguous)
------------------------------------------------------------------------------
  1. Trading year length: Milestone-Eight specifies 365 days. We use
  2024-01-01 to 2024-12-30 (2024 is technically a leap year, but the spec
  explicitly says 365 days so we honour that).

  2. Subscription tier user limits: Project Requirements Docket Rule #2
  mentions 'a limit of xx users' but does not specify exact numbers per
  tier. We assigned: Free=10, Freemium=25, Premium=50, Enterprise=200
  (reasonable tiered capacity that comfortably accommodates 300 users
  across 30 clients at 10 users/client avg).

  3. Subscription monthly cost: Not specified in milestone docs. Assigned:
  Free=£0, Freemium=£49, Premium=£199, Enterprise=£499.

  4. Expense distribution: Milestone-Eight specifies upper bounds
  (vendors<£200, bills/wage<£1,000, etc.) but not the distribution shape.
  Used exponential distribution (mode near 0, capped at upper bound) to
  keep daily expenses realistic and EOD positive on most days. Uniform
  distribution would have driven EOD negative too often.

  5. Repairs 'occasionally exceed £1,000' (Milestone-Eight): Interpreted
  as 5% probability of £1,000-£1,500.

  6. Banking when EOD < £10: Spec says 'between £10 and the full EOD'.
  When EOD is below £10, we bank the full EOD (or 0 if EOD ≤ 0) — the spec
  doesn't define this edge case explicitly.

  7. 'wage advances' column: Milestone-Eight mentions 'wage, advances' as
  2 separate items but the existing Expenses schema has them combined as
  'wage advances' (one column). Preserved the existing column to avoid
  schema drift; treated as one component. Total expense = bills + vendors
  + wage_advances + repairs + sundries.

  8. sealed_by in Banking: Spec says 'a generated staff name, linked to
  the restaurant's user pool'. Stored as a staff name (text) drawn from
  the user pool of the restaurant's country. Soft link, not a hard FK.

  9. Cash_Up.match_status: Spec doesn't explicitly define this column's
  meaning. Interpreted as internal reconciliation flag (BOD + sales -
  expenses - delivery_charges == EOD), which holds TRUE by construction.

  10. Deliveries.match_status distribution: Milestone docs define
  match_status = TRUE if order_amount == api_amount else FALSE, and state
  'Any discrepancy is flagged as a mismatch requiring investigation.' The
  docs do NOT specify a target distribution. Per user instruction,
  enforced 95% TRUE (208,050 rows) / 5% FALSE (10,950 rows) — the 5%
  mismatches represent real reconciliation anomalies that downstream
  workflows investigate. Each FALSE row is traceable by delivery_id with a
  discrepancy of £0.50-£5.00.
